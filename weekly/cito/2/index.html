a unit of work

Double-entry bookkeeping is a system for achieving balanced transaction records, by which every entry to an account requires a corresponding and opposite entry to a different account. 

twashing/bkell

Bkell is an open source Clojure project providing a shell and API for maintaining these balanced transaction records. It stores them using Datomic, and currently lacks functionality to import a dataset into it.

Your job is to write bkell’s import function. The specs:

  • qualified name: bkell.bkell/import
  • arguments: 
    1. data: a simple map tree, of the data that a user intends to import to their data store.
    2. group: the group into which the data will be imported.
  • expected return: nil or an exception

For further information, please consult the GitHub issue:

Implement Import Function (#1)

Clojurist: Eric Normand

Eric is the editor of the Clojure Gazette and creator of LispCast, and this is Clojure in the Open's brief interview with him, inspired by the usesthis.com format.

Eric, what do you do?

I help people thrive with Clojure. That might mean the company I work for (companies are groups of people), my readers, my video viewers, people I meet, etc. Clojure has a lot of value to bring to software engineering, and I want to see it spread. At the heart, there are always people. More concretely, I write the Clojure Gazette, I make LispCast videos and I blog. I work at BuyHappy.

What drew you into Clojure and what made you stay?

I've been into Lisp and functional programming for over a decade. When Clojure came out, I was skeptical at first (another Lisp?) but I soon saw how it captured a lot of the best of the older Lisps but lots of modern ideas as well. I just never looked back.

What's your workflow? 

I do most of my work in Emacs. I like to keep the setup relatively vanilla. Life is too short to be configuring all the time. I keep the configuration in Dropbox and symlink to it so I can have the same one on all my machines.

I use this thing called Phoenix on my Mac (work computer) so I don't have to resize windows with the mouse. It's the most minimal tiling setup. You can put two windows side-by-side with a few easy keystrokes. I'm also emailing a lot, and though I've tried to find a better client, I keep coming back to the GMail web client. With all its faults, it just works.

For coding, I use Cider. Once you get it working, it works well. Leiningen is always there. I use Heroku a lot for work and side projects. The Clojure Gazette is published using MailChimp.

For video, I use so many things, and it's constantly evolving. I write the code that I think someone should be able to write at the end of the video. Then I break it down into its individual learning components. I call that the curriculum. They need to be able to use if statements, write functions, and build maps. Then I try to find motivation for each of those things. Even a small motivation helps. Even a fictional motivation is real to the brain. Video is engaging.

Then I write the script and the slides in tandem. The script is in Emacs and I do the visuals in Keynote. Keynote doesn't have a huge variety of animations, but it forces me to keep it simple. I then record the script in Audacity. I record everything in Screenflow. Then I edit in Final Cut Pro.

What's the biggest pain point or what would be your ideal workflow?

The computer interface is the pain point. I would love for computers to disappear. Not go away, but just become part of everything. My ideal would be smart paper, where I could write out an email longhand, draw a mail icon at the bottom, and it would send it. Then I crumple it up or file it away. Paper still has advantages, so we keep coming back to it. We should just merge them together.

In terms of programming, it's similar. I really wish that programming were more like painting or sculpture. I don't paint that well, nor do I sculpt, but I can watch someone do their art and at least appreciate it. It seems so direct, and the skill is apparent. I cannot share programming with my non-programmer friends because it's both foreign to them and private. It is much more like poetry. Watching a poet write is not instructive.

For editing video, my ideal workflow would mean working in the final form faster. So imagine Keynote (like PowerPoint) where you edit the slide, and what you see is what you get. But why can't that be a video, and I just say stay on this slide for 3 seconds, then fade to this code being typed (and I can change the code right there). Instead, I have to do the slides first, then record the video. Once it's video, all I can do is play with time (speed it up, cut something out, etc). If I need to make a change, I need to go back to Keynote and record again. Basically wysiwyg for exactly my style of video.

That said, I am still experimenting with the video style, and I'm afraid it's just too time consuming for the value. The videos are good, but I'd like to be able to do them faster. I'll probably switch to live action video recording, which has its problems, but will be way more expressive for less effort.
